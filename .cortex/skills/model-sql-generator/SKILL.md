# Model SQL Generator

Generates a dbt model SQL file with CTE-based transformation logic derived from the contract's column definitions and derivation rules. This is an **AI-powered** skill — it interprets business logic from the contract to write transformation SQL.

## When to Use

- Phase 3: Deliver — generating the core dbt model
- When the user asks to generate or regenerate the transformation SQL
- As part of the `$data-product-generator` orchestration (Step 2)

## Inputs

- **Contract YAML**: parsed via `$contract-parser`
- **Source table schemas**: from Snowflake (to verify column availability)

## Workflow

### Step 1: Parse the Contract

Read the contract YAML and extract:
- Source tables with their columns and filters
- Output column definitions with derivation logic
- Business rules that imply transformation logic
- Primary key and score/category columns

### Step 2: Verify Source Columns Exist

```sql
-- For each source table:
DESCRIBE TABLE {database}.{schema}.{table};
```

Confirm that all columns referenced in derivation logic actually exist in source tables.

### Step 3: Generate the CTE Pipeline

Follow the CTE-based transformation pattern:

```sql
{{
  config(
    materialized='table',
    tags=[{tags_from_contract}],
    meta={
      'owner': '{owner}',
      'contract_version': '{version}',
      'description': '{description}'
    }
  )
}}

-- Source CTEs (one per upstream table with filters)
WITH source_{entity_1} AS (
    SELECT {columns}
    FROM {{ source('{raw_schema}', '{table}') }}
    WHERE {filter_from_contract}
),

-- Aggregation CTEs (group by primary entity)
{entity}_aggregates AS (
    SELECT {primary_key},
      {aggregation_logic_from_derivation}
    FROM source_{entity}
    GROUP BY {primary_key}
),

-- Signal/Feature CTEs (boolean flags, engagement scores)
{entity}_signals AS (
    SELECT *,
      {flag_logic_from_business_rules}
    FROM {previous_cte}
),

-- Scoring CTE (composite score calculation)
scored AS (
    SELECT *,
      {scoring_formula_from_derivation}
    FROM {previous_cte}
),

-- Final CTE (tier assignment, metadata)
final AS (
    SELECT
      {all_output_columns},
      CURRENT_TIMESTAMP() AS score_calculated_at
    FROM scored
)

SELECT * FROM final
```

**Key rules for model generation:**
- Use `{{ source('schema', 'table') }}` for source references
- Use `{{ config(materialized='table') }}` at the top
- Do NOT include `schema=` in the config block
- Use `LEAST()` / `GREATEST()` for capping scores to valid ranges
- Build JSON objects with `OBJECT_CONSTRUCT()` for complex driver columns
- Follow the derivation logic from the contract exactly

### Step 4: Present to User

-> ASK USER: "Generated dbt model with:
  - [N] CTEs in pipeline
  - [N] output columns
  - Scoring formula: {brief description}
  - [N] lines of SQL
  
  Review the model? Any changes needed?"

### Step 5: Save the File

Save to `03_deliver/dbt_project/models/{model_name}.sql`

Add header comment:
```sql
-- Generated from: 02_design/{contract_name}.yaml
-- Owner: {owner}
-- Version: {version}
-- [INTERVENTION] YYYY-MM-DD: Generated by $model-sql-generator
```

## Output

- `03_deliver/dbt_project/models/{model_name}.sql` — Complete dbt model SQL

## Guardrails

- Every output column MUST trace back to a contract `schema.properties` entry
- NEVER include `schema=` in the dbt config block (causes name concatenation)
- NEVER use external packages (dbt_utils, etc.)
- ALWAYS use `LEAST()` / `GREATEST()` to cap scores within contract-defined ranges
- ALWAYS include `meta` tags with owner, contract_version, description
- ALWAYS add `tags` from the contract's `info.tags`
- Use `IS_ROLE_IN_SESSION()` if any role checks are needed (never `CURRENT_ROLE()`)
- Score/range columns must respect the contract's `minimum` and `maximum` values
