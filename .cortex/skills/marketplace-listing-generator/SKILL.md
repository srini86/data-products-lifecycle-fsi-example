# Marketplace Listing Generator

Generates Snowflake DDL for an internal marketplace listing: secure share, listing metadata, and consumer role grants. This is a **template-based** skill that parses the contract's stakeholder and access sections.

## When to Use

- Phase 3: Deliver — making the data product discoverable internally
- When the user asks to publish a data product to the internal marketplace
- As part of the `$data-product-generator` orchestration (Step 8)

## Inputs

- **Contract YAML**: parsed via `$contract-parser` — specifically the `stakeholders` and `access` sections
- **Target table**: fully qualified name

## Workflow

### Step 1: Extract Listing Metadata from Contract

From the contract, extract:
- Product title and description (from `info`)
- Stakeholders/consumers (from `stakeholders`)
- Access tiers (from `access`) — which roles get what level of access
- Tags and categories

### Step 2: Confirm Listing Details with User

-> ASK USER: "I'll create an internal marketplace listing:
  - Share name: {DATA_PRODUCT_NAME}_SHARE
  - Title: {title}
  - Consumer roles: {list}
  - Access tiers: {summary}
  
  Proceed? Any changes to the listing metadata?"

### Step 3: Generate Marketplace DDL

```sql
-- ============================================================================
-- INTERNAL MARKETPLACE LISTING: {data_product_name}
-- ============================================================================
-- Generated from: {contract_path}
-- Contract: {name} v{version}
-- [INTERVENTION] YYYY-MM-DD: Generated by $marketplace-listing-generator
-- ============================================================================

-- =============================================================================
-- STEP 1: CREATE SECURE SHARE
-- =============================================================================

CREATE OR REPLACE SHARE {share_name}
  COMMENT = '{description}';

-- Grant usage on database and schema
GRANT USAGE ON DATABASE {database} TO SHARE {share_name};
GRANT USAGE ON SCHEMA {database}.{schema} TO SHARE {share_name};

-- Grant SELECT on the data product table
GRANT SELECT ON TABLE {database}.{schema}.{table} TO SHARE {share_name};

-- Grant SELECT on the semantic view (if exists)
GRANT SELECT ON VIEW {database}.{schema}.{model_name}_sv TO SHARE {share_name};

-- =============================================================================
-- STEP 2: CREATE LISTING
-- =============================================================================

CREATE OR REPLACE LISTING {listing_name}
  IN DATA EXCHANGE {exchange_name}
  AS $$
  title: "{title}"
  description: "{description}"
  terms_of_service:
    type: "OFFLINE"
  targets:
    accounts:
      - "{consumer_account}"
  $$;

ALTER LISTING {listing_name}
  SET SHARE = {share_name};

-- =============================================================================
-- STEP 3: CONSUMER ROLE GRANTS
-- =============================================================================
-- Define roles for different access tiers from the contract

{for_each_role}
CREATE ROLE IF NOT EXISTS {consumer_role};
GRANT SELECT ON TABLE {database}.{schema}.{table} TO ROLE {consumer_role};
-- Note: Masking policies enforce column-level access per role
{end_for}
```

### Step 4: Save the File

Save to `03_deliver/dbt_project/marketplace_listing.sql`

## Output

- `03_deliver/dbt_project/marketplace_listing.sql` — Share, listing, and role DDL

## Guardrails

- ALWAYS confirm with the user before generating share/listing DDL
- NEVER create shares to external accounts without explicit user approval
- Include the semantic view in the share if one exists
- Consumer roles should align with the masking policy authorized roles from the contract
- Share naming convention: `{DATA_PRODUCT_NAME}_SHARE`
- Include comments in the share for discoverability
- If the contract has no stakeholders section, ask the user who the consumers are
