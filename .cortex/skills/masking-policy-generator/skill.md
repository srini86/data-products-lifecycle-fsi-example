# Masking Policy Generator

Generates Snowflake masking policy DDL by parsing the contract's PII column definitions and authorized roles. This is a **template-based** skill — deterministic output from contract data.

## When to Use

- Phase 3: Deliver — generating governance artifacts
- When the user asks to generate or regenerate masking policies
- As part of the `$data-product-generator` orchestration (Step 5)

## Inputs

- **Contract YAML**: parsed via `$contract-parser` — specifically the `access` section and columns with `pii: true`

## Workflow

### Step 1: Extract PII Columns from Contract

For each column with `pii: true`, extract:
- Column name
- Data type (STRING, NUMBER, etc.)
- Masking policy name (from `masking_policy.name`)
- Authorized roles (from `masking_policy.authorized_roles`)
- Masking behavior (full mask, partial mask, etc.)

### Step 2: Confirm Authorized Roles with User

-> ASK USER: "The contract defines these masking policies:

| Column | Policy Name | Authorized Roles | Default Mask |
|--------|-------------|-----------------|--------------|
| {col} | {policy} | {roles} | ***MASKED*** |

Are the authorized roles correct? Any roles to add or remove?"

Wait for approval before generating.

### Step 3: Generate Masking Policy DDL

For each PII column:

```sql
-- ============================================================================
-- MASKING POLICY: {POLICY_NAME}
-- ============================================================================
-- Applies to: {column_name}
-- Authorized roles: {role_list}
-- ============================================================================

CREATE OR REPLACE MASKING POLICY {database}.{governance_schema}.{policy_name}
AS (val {data_type})
RETURNS {data_type} ->
    CASE
        WHEN IS_ROLE_IN_SESSION('{role_1}') THEN val
        WHEN IS_ROLE_IN_SESSION('{role_2}') THEN val
        ELSE '***MASKED***'
    END;
```

Then generate the ALTER statements to apply policies to columns:

```sql
-- ============================================================================
-- APPLY MASKING POLICIES TO TABLE
-- ============================================================================

ALTER TABLE {database}.{schema}.{table}
  MODIFY COLUMN {column}
  SET MASKING POLICY {database}.{governance_schema}.{policy_name};
```

### Step 4: Save the File

Save to `03_deliver/dbt_project/masking_policies.sql` (or `03_deliver/masking_policies.sql`)

Add header comment:
```sql
-- Generated from: {contract_path}
-- Contract: {name} v{version}
-- [INTERVENTION] YYYY-MM-DD: Generated by $masking-policy-generator
```

## Output

- `03_deliver/dbt_project/masking_policies.sql` — Masking policy DDL + ALTER statements

## Guardrails

- ALWAYS use `IS_ROLE_IN_SESSION()` — NEVER `CURRENT_ROLE()` (fails with role hierarchy)
- ALWAYS confirm authorized roles with the user before generating
- Policy naming convention: `{COLUMN}_MASK` (e.g., `EMAIL_ADDRESS_MASK`)
- Default masked value: `'***MASKED***'` unless the contract specifies otherwise
- Generate in the GOVERNANCE schema (not DATA_PRODUCTS)
- Include both CREATE POLICY and ALTER TABLE SET MASKING POLICY statements
- NEVER apply masking policies without user confirmation
