# DMF Setup Generator

Generates Snowflake Data Metric Function (DMF) setup SQL by mapping contract quality rules to built-in DMF functions. This is a **template-based** skill — deterministic mapping from rule type to DMF.

## When to Use

- Phase 3: Deliver — generating data quality monitoring artifacts
- When the user asks to generate or regenerate DMF setup
- As part of the `$data-product-generator` orchestration (Step 6)

## Inputs

- **Contract YAML**: parsed via `$contract-parser` — specifically the `quality_rules` section

## Workflow

### Step 1: Map Quality Rules to DMFs

| Contract Rule Type | DMF Function |
|-------------------|-------------|
| `completeness` (required columns) | `SNOWFLAKE.CORE.NULL_COUNT` |
| `uniqueness` (primary key) | `SNOWFLAKE.CORE.DUPLICATE_COUNT` |
| `uniqueness` (business keys) | `SNOWFLAKE.CORE.UNIQUE_COUNT` |
| `row_count` | `SNOWFLAKE.CORE.ROW_COUNT` |
| `freshness` (SLA) | `SNOWFLAKE.CORE.FRESHNESS` |

### Step 2: Generate DMF Setup SQL

```sql
-- =============================================================================
-- DATA METRIC FUNCTIONS (DMF) SETUP: {data_product_name}
-- =============================================================================
-- Generated from: {contract_path}
-- Contract: {name} v{version}
-- [INTERVENTION] YYYY-MM-DD: Generated by $dmf-setup-generator
-- =============================================================================

USE DATABASE {database};
USE SCHEMA {schema};

-- Grant DMF permissions
GRANT EXECUTE DATA METRIC FUNCTION ON ACCOUNT TO ROLE {role};

-- Set schedule
ALTER TABLE {table}
    SET DATA_METRIC_SCHEDULE = 'TRIGGER_ON_CHANGES';

-- =============================================================================
-- COMPLETENESS CHECKS (NULL_COUNT)
-- =============================================================================
-- Contract Rule: {completeness_rule_name}

ALTER TABLE {table} ADD DATA METRIC FUNCTION
    SNOWFLAKE.CORE.NULL_COUNT ON ({column});
-- Repeat for each required column

-- =============================================================================
-- UNIQUENESS CHECKS (DUPLICATE_COUNT)
-- =============================================================================
-- Contract Rule: {uniqueness_rule_name}

ALTER TABLE {table} ADD DATA METRIC FUNCTION
    SNOWFLAKE.CORE.DUPLICATE_COUNT ON ({primary_key});

-- =============================================================================
-- ROW COUNT
-- =============================================================================

ALTER TABLE {table} ADD DATA METRIC FUNCTION
    SNOWFLAKE.CORE.ROW_COUNT ON ({table});

-- =============================================================================
-- FRESHNESS CHECK
-- =============================================================================
-- Contract SLA: {freshness_hours}h freshness

ALTER TABLE {table} ADD DATA METRIC FUNCTION
    SNOWFLAKE.CORE.FRESHNESS ON ({timestamp_column});
```

### Step 3: Present to User

-> ASK USER: "Generated DMF setup with:
  - [N] NULL_COUNT checks (required columns)
  - [N] DUPLICATE_COUNT checks (primary key)
  - ROW_COUNT metric
  - FRESHNESS metric on {timestamp_column}
  - Schedule: TRIGGER_ON_CHANGES
  
  Total: [N] data metric functions. Review?"

### Step 4: Save the File

Save to `03_deliver/dbt_project/dmf_setup.sql`

## Output

- `03_deliver/dbt_project/dmf_setup.sql` — DMF configuration SQL

## Guardrails

- Schedule MUST be `TRIGGER_ON_CHANGES` (not time-based)
- ALWAYS include NULL_COUNT for every column marked `required` in the contract
- ALWAYS include DUPLICATE_COUNT on the primary key column
- ALWAYS include ROW_COUNT and FRESHNESS
- Use fully qualified function names: `SNOWFLAKE.CORE.{function}`
- Include the GRANT statement for DMF execution permissions
- Do NOT create custom DMFs — use only the built-in `SNOWFLAKE.CORE.*` functions
