# Semantic View Generator

Generates a Snowflake Semantic View DDL that enables natural language queries via Cortex Analyst. Parses the contract to define dimensions and metrics, excluding PII columns. This is a **template-based** skill.

## When to Use

- Phase 3: Deliver — generating the semantic layer
- When the user asks to create a semantic view for Cortex Analyst
- As part of the `$data-product-generator` orchestration (Step 7)

## Inputs

- **Contract YAML**: parsed via `$contract-parser` — column definitions and PII flags
- **Target table**: fully qualified name

## Workflow

### Step 1: Classify Columns

From the contract, classify each output column:

| Classification | Criteria | Semantic View Role |
|---------------|----------|-------------------|
| PII column | `pii: true` | EXCLUDE from semantic view |
| Categorical | type is STRING with enum values | DIMENSION |
| Numeric (aggregatable) | type is NUMBER/INTEGER, is a score or count | FACT or METRIC |
| Identifier | primary key or foreign key | DIMENSION |
| Timestamp | type is TIMESTAMP | DIMENSION (for time-based analysis) |
| Boolean/Flag | type is BOOLEAN | DIMENSION |

### Step 2: Generate Semantic View DDL

```sql
-- ============================================================================
-- SEMANTIC VIEW: {data_product_name}_sv
-- ============================================================================
-- Enables natural language queries via Cortex Analyst
-- Generated from: {contract_path}
-- Contract: {name} v{version}
-- NOTE: PII columns excluded for safe self-service analytics
-- [INTERVENTION] YYYY-MM-DD: Generated by $semantic-view-generator
-- ============================================================================

CREATE OR REPLACE SEMANTIC VIEW {database}.{schema}.{model_name}_sv
TABLES (
    t AS {schema}.{table} PRIMARY KEY ({primary_key})
)
DIMENSIONS (
    -- Identifiers
    t.{primary_key} AS {primary_key}
      COMMENT '{description}',

    -- Categorical dimensions
    t.{category_column} AS {category_column}
      COMMENT '{description}',

    -- Timestamp dimensions
    t.{timestamp_column} AS {timestamp_column}
      COMMENT '{description}',

    -- Boolean flag dimensions
    t.{flag_column} AS {flag_column}
      COMMENT '{description}'
)
FACTS (
    -- Numeric measures
    t.{score_column} AS {score_column}
      COMMENT '{description}'
)
METRICS (
    -- Pre-defined aggregations
    AVG(t.{score_column}) AS avg_{score_column}
      COMMENT 'Average {description}'
);
```

### Step 3: Present to User

-> ASK USER: "Generated semantic view '{model_name}_sv' with:
  - [N] dimensions (categorical, timestamp, boolean)
  - [N] facts (numeric measures)
  - [N] metrics (pre-defined aggregations)
  - [N] PII columns excluded: {list}
  
  Review? Any columns to add or remove?"

Save to `03_deliver/dbt_project/semantic_view.sql`

## Output

- `03_deliver/dbt_project/semantic_view.sql` — Semantic View DDL

## Guardrails

- NEVER include PII columns (those with `pii: true`) in the semantic view
- ALWAYS include a COMMENT for every dimension, fact, and metric
- ALWAYS set the PRIMARY KEY in the TABLES clause
- Use meaningful metric names (e.g., `avg_churn_risk_score` not just `avg_score`)
- Comments should be natural-language descriptions suitable for LLM interpretation
- If the table has no numeric columns suitable for metrics, create at least a COUNT metric
